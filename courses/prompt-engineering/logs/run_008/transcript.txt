**Lesson 2: Role assignment**

Giving the model a role — "You are a backend architect" — narrows how it thinks about the task: the tone, the depth, the assumptions it defaults to. The rule is **Role-before-task**: the role sentence must come *before* the task description in your prompt, never tacked onto the end. When the model reads the role first, every word that follows is filtered through that persona. If you put the role last, the model has already processed the task neutrally and only retroactively tries to "act as" someone — a weaker effect.

MENTOR: Here we go — Lesson 4: **Chain-of-thought**.

A generic "think step by step" instruction reduces jump-to-conclusion errors on multi-step tasks, but it has a blind spot: the model often blurts out a conclusion-flavored first sentence and then backfills reasoning to match. The "Chain from data, not from conclusion" principle fixes that by forcing the model to enumerate the *given facts first*, derive intermediate conclusions from those facts, and produce the final answer *only* as the last line — the answer is not allowed to appear before the reasoning that produced it. In practice, that means an explicit "step 0: list the input data" and a hard rule that the verdict must wait until the final line.

So: **what specific failure does "Chain from data, not from conclusion" prevent that a plain "think step by step" does not?**

**Lesson 7: Iterative Refinement**

Our term for this is **"Prompt-generating-prompts"**: a meta-prompt takes a small set of parameters (like task type, audience, constraints) as input and outputs a fully-formed, directly-usable prompt — one that itself follows the 1-task-1-verb rule from L1. The output must be a ready-to-use prompt, not a description of one or a template with blanks still in it. You validate a meta-prompt by running its *output* through the earlier course rules as a checklist — if the generated prompt fails those checks, the meta-prompt needs fixing.
